# Session 07: Stream Preview Resize Bug — Resolved, Plus Title Overlay Polish

Date: 2026-07-04

Continuation of the feature/bug started in [session-06-stream-preview-thumbnail-and-resize-bug.md](session-06-stream-preview-thumbnail-and-resize-bug.md). That session ended with an *unresolved* real-app bug (thumbnail overflow/distortion on window resize) and a sizeHint fix that had been applied but not proven to address the actual symptom. This session found the real root cause, fixed it, verified the fix with a genuine reproduction, and then did a UI polish pass on the stream-title caption.

## What Happened

1. Resumed via plan mode. Used an Explore agent to map the exact real widget nesting (`gui_qt/main_window.py`'s `QGridLayout`, 2:3 column stretch, no `QSplitter`) and a Plan agent to design a fix for the (at the time) leading theory: `_LandscapePreviewLabel` inheriting `QLabel`'s default `sizeHint()`/`minimumSizeHint()` (which track the currently-held pixmap), feeding back into the grid's column-width negotiation during a drag.
2. Implemented that fix (override `sizeHint()`/`minimumSizeHint()` to derive from a constant `minimumWidth()`, not the pixmap). Before trusting it, built five increasingly faithful offscreen reproductions (bare stub sibling → real `FavoritesPanel` → full real hierarchy including `StreamActions`/`StatusDisplay` → real `showMaximized()` call). **None reproduced the reported bug, with or without the fix** — the 2:3 column split held steady in every case. Kept the sizeHint fix anyway (it's a legitimate anti-pattern removal) but flagged it as unconfirmed.
3. User re-tested the real app: **no visible change**, bug persists exactly as before ("goes beyond the square and part of the preview image is being cut off" when maximized).
4. Added **temporary debug logging** (`[PREVIEW-DEBUG]` prefix, `logger.info()` so it shows with default app settings) to `_LandscapePreviewLabel.resizeEvent`/`_sync_height`/`_apply_scaled_pixmap`/`paintEvent` and `ChatPanel.resizeEvent`, capturing widget sizes, pixmap sizes, and window/maximized state at every frame.
5. User ran the real app, reproduced the bug, and shared the log (966 `[PREVIEW-DEBUG]` lines). **The log showed no anomaly at all** — every `paintEvent` had the held pixmap matching the label's own rect almost exactly, and `ChatPanel` never exceeded its 60% grid column share. Confirmed via `AskUserQuestion` that the bug **persists** in the settled maximized state (not a transient drag-animation artifact) — ruling out a "live-resize rendering race" explanation too.
6. User shared **screenshots** (non-maximized vs. maximized). This is what actually broke the case open: in the maximized screenshot, the "Activity" section (`StatusDisplay`, grid row 2) was **entirely absent** from the visible window — a vertical layout problem, not the horizontal/column-width problem every prior theory assumed.
7. **Root cause found and confirmed**: `_LandscapePreviewLabel._sync_height()` computed height purely as `round(self.width() / (16/9))`, with zero regard for how much vertical room `ChatPanel` actually had. At the real app's logged maximized size (1920×1009), this demanded ~620px of height for the image alone — more than `ChatPanel`'s own allotted height (761px) could spare once its buttons/labels/separators took their share. The image's bottom edge extended ~140px past `ChatPanel`'s own bottom edge and got clipped there (Qt widgets clip child rendering to their own contentsRect). Reproduced this **deterministically** offscreen using the real widget hierarchy at the exact log dimensions (902×823 → 1920×1009): with the cap disabled, label bottom = 901px vs. `ChatPanel` height = 761px (140px overflow, confirmed via direct geometry assertion).
8. **Fix implemented**: added `_LandscapePreviewLabel.set_max_height(max_height)` (caps the width-driven height to an externally supplied ceiling) and `ChatPanel._update_preview_max_height()` (sums the `sizeHint().height()` of every *other* item in `ChatPanel`'s own `QVBoxLayout`, plus inter-item spacing, subtracts that from `self.contentsRect().height()`, and passes the remainder to the label) — computed fresh every resize, not a hardcoded pixel value, per the user's explicit preference against hardcoding. Called from both `ChatPanel.__init__` and `ChatPanel.resizeEvent()`.
9. **Verified properly**: re-ran the exact same real-hierarchy-at-exact-log-dimensions reproduction with the fix restored — label bottom now 713px, within `ChatPanel`'s 761px. Added this as a permanent test, `tests/test_chat_panel_preview_resize.py::test_preview_image_does_not_overflow_chat_panel_on_wide_maximized_window`, and confirmed (by temporarily disabling the fix) that it **fails without the fix and passes with it** — a real, proven-effective regression test, unlike the earlier sizeHint test which never actually distinguished fixed/unfixed code in a realistic scenario.
10. Removed **all** temporary `[PREVIEW-DEBUG]` logging — none remains in the shipped code. Full suite: 165 passed (164 prior + 1 new; same 2 pre-existing unrelated stray-module test failures, excluded via `--ignore`).
11. **UI polish requests** (after the fix was confirmed usable): user asked for the stream title text to be bolder with a shadow for readability. Added `font-weight: 600` to `QLabel#streamPreviewTitle` in both `dark.qss`/`light.qss`, and a `QGraphicsDropShadowEffect` applied in code (Qt stylesheets don't support `text-shadow`).
12. **Follow-up UI request**: user said they'd prefer the title overlaid *on top of* the preview image rather than taking its own row below it, since the image is "more important than the title" and the title's own row was eating into the image's (already-limited) vertical budget. Reworked `_LandscapePreviewLabel` to own a `title_label` as a **child widget** (not layout-managed), positioned via manual `setGeometry()` at the bottom of the image, sized via `heightForWidth()` for correct word-wrap height, auto-hidden when there's no title text. Removed the title label from `ChatPanel`'s own `QVBoxLayout` entirely — this also means `_update_preview_max_height()`'s "other height" sum (built in step 8) automatically excludes it now, so the image gets *more* room than before, which was the point. Restyled the overlay as white text on a `rgba(0,0,0,150)` semi-transparent bar (theme-independent now, since it sits on photo content, not app chrome) with the same bold weight + drop shadow.

## Key Decisions

- **Trust real data over plausible-sounding theories for GUI layout bugs.** Two theories in a row (sizeHint feedback loop; stale-pixmap-scaling mismatch) were reasoned through carefully and even had partial supporting logic, but neither held up against real logs/screenshots. The eventual fix only came from getting actual log data + screenshots from the user's real running app. **Lesson for future sessions**: when an offscreen repro won't reproduce a GUI bug after 2+ genuine attempts, stop theorizing and get real data (logs, screenshots) before implementing another fix.
- Kept the sizeHint()/minimumSizeHint() override even though it didn't fix the reported bug — it's independently correct (a widget shouldn't report a preferred size that tracks its own displayed content) and is covered by its own regression test.
- Available-height computation for the image is derived dynamically from sibling `sizeHint()`s each resize, not a hardcoded pixel cap — consistent with the user's stated dislike of hardcoded sizes (same principle from the original responsive-sizing request in session 06).
- Title overlay uses manual child-widget positioning (not a layout) since it needs to sit *on top of* the image rather than take its own row — this is a deliberate, minimal way to overlay content in Qt without restructuring the whole panel into a `QStackedLayout`.
- Debug logging was temporary by design (clearly marked, prefixed, and fully removed once the real root cause was found) — this turned out to be the right call methodologically: it was ultimately the *screenshots*, not the log data, that solved it, but the log data was still useful for ruling out the wrong theories first (confirming pixmap/rect always matched, confirming ChatPanel never exceeded its grid share).

This is a rendering environment limitation, not a code issue, but it does mean the visual result of the title-overlay change has **not yet been eyeballed in the real running app**.

## Things Not Yet Tried / Open Follow-ups

Live confirmation above covered general usability (resize/maximize behavior, overlay legibility) but not necessarily every edge case below — these remain untested unless it comes up again:

1. The overlay's bottom corners are square while the image's box has `border-radius: 4px` — if this looks visually off in the corners, consider adding a matching `border-radius` to the bottom corners of `QLabel#streamPreviewTitle`, or inset the overlay slightly from the very edges.
2. Long titles: `title_label` word-wraps via `heightForWidth()`, but there's no cap on how many lines it can grow to — a very long title could visually dominate a short/small preview image. Not yet stress-tested with a very long real Twitch title.
3. The available-height computation in `ChatPanel._update_preview_max_height()` sums sibling `sizeHint()`s, which is an approximation (particularly for word-wrapped labels, whose `sizeHint()` doesn't account for actual wrapped height at a constrained width) — this is a deliberately conservative approximation, not pixel-perfect, and hasn't been stress-tested against extreme window aspect ratios beyond the one real scenario from the log (1920×1009).
4. User mentioned in passing that ideally the whole GUI layout could be restructured to give the preview image more room, but explicitly said this isn't something they care about right now — noted here in case it resurfaces later, not an active task.
