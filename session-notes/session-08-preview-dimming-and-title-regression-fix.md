# Session 08: Preview Dimming While Watching + Title-Overlay Regression Fix

Date: 2026-07-04

Continuation of the preview-thumbnail feature from [session-06-stream-preview-thumbnail-and-resize-bug.md](session-06-stream-preview-thumbnail-and-resize-bug.md) and [session-07-preview-resize-fix-and-title-overlay.md](session-07-preview-resize-fix-and-title-overlay.md). Both of those sessions' fixes (resize/overflow fix, title overlay) were confirmed working live by the user at the end of session 07. This session added a new polish feature (dim the preview while a stream plays) and then found + fixed a real regression that feature introduced (the title overlay silently stopped rendering its background/text).

## What Happened

1. **New feature requested**: when the user opens/watches a stream, the preview thumbnail + title should stay visible but desaturate (grayscale) and fade (reduced opacity) instead of disappearing entirely - so it doesn't compete for attention with the actual video - and return to normal once the stream stops.
2. Investigated the existing flow: `ChatPanel.set_streaming(active)` is already called at exactly the right moments (`gui_qt/stream_gui.py`'s `_on_stream_started`/`_on_stream_finished`/`_on_stream_error`), and previously only toggled the clip button's enabled state. Also found that `_on_stream_started` was calling `self.chat_panel.clear_preview()` on stream start, which is *why* the preview disappeared entirely at the time - that call was removed so the existing preview persists (to then be dimmed) instead of being wiped.
3. **First implementation** (later found to be buggy): added a `QGraphicsOpacityEffect` installed directly on `_LandscapePreviewLabel` in `__init__` (always present, opacity toggled between 1.0 and 0.55 via a new `set_dimmed()` method), plus grayscale pixmap conversion. Wired `ChatPanel.set_streaming()` to call `set_preview_dimmed(active)`. Verified via an offscreen script that opacity and grayscale toggled correctly, and the full test suite (165 tests) passed. Reported this as done.
4. **User reported a regression**: "I'm no longer seeing the stream title" after this change. Investigated via an offscreen repro *with the real theme QSS actually loaded* (an earlier verification script had NOT loaded the app's stylesheet, which is why it hadn't already caught this) and found the title overlay's background rendered as fully transparent - showing the raw image color through it - **even at opacity 1.0, before any dimming had ever been triggered**. So the bug existed from the moment `ChatPanel` was constructed, independent of the dimming feature actually firing.
5. **Root cause confirmed**: `title_label` (the caption overlay) has its own `QGraphicsDropShadowEffect`. Installing a *second*, separate `QGraphicsEffect` (the new opacity effect) on its *parent* widget (`_LandscapePreviewLabel`) suppresses the child's own stylesheet-painted background - this is a known Qt quirk/limitation where a parent widget's `QGraphicsEffect` breaks correct rendering of a child that has its own independent `QGraphicsEffect`. Confirmed by removing the parent effect in a scratch script and observing the title's background render correctly again.
6. **Fix**: reworked dimming to avoid `QGraphicsEffect` on the widget entirely.
   - Image dimming: `_LandscapePreviewLabel._dim_pixmap()` bakes both desaturation (`QImage.Format_Grayscale8` conversion) and fade (via `QPainter.setOpacity()` while compositing onto a transparent `QPixmap`) directly into the pixmap's pixel data before `setPixmap()`.
   - Title dimming: `set_dimmed()` now toggles `title_label.setStyleSheet(...)` with an inline override (`background-color: rgba(0, 0, 0, 90); color: rgba(255, 255, 255, 140);`) instead of a second effect - this only overrides those two properties per-instance while padding/font-size/weight still cascade from the app-level `QLabel#streamPreviewTitle` QSS rule.
7. Verified the fix offscreen with the real theme QSS loaded: title stays visible with its darkened background bar intact through a dim -> undim cycle; image pixel goes fully grayscale (`r == g == b`) while dimmed and restores to pure color afterward.
8. **User explicitly asked for a regression test** so this can't silently reoccur. Added `tests/test_chat_panel_preview_dimming.py` with the real dark theme QSS loaded (necessary since the bug is specifically about a QSS-styled property being suppressed - a bare `QApplication` with no stylesheet wouldn't have exposed it, which is exactly how the first, buggy implementation slipped through verification).
9. **Proved the new test actually catches the regression**: rather than repeat the earlier `git stash` mistake (see [session-07](session-07-preview-resize-fix-and-title-overlay.md)'s "Errors and fixes" - stashing a whole file reverts *all* uncommitted changes to it, not just the latest edit), this time the old buggy `QGraphicsOpacityEffect`-on-parent approach was reproduced in a standalone scratch script that imported the real `ChatPanel`/`_LandscapePreviewLabel` classes and manually attached the old-style effect, without editing any tracked file. Confirmed the title-bar pixel comes back as pure red (`(255, 0, 0)`, i.e. no background painted) under that old approach - exactly the condition `test_title_overlay_stays_visible_and_styled_through_dim_cycle`'s first assertion catches.
10. Note: `git stash push -- gui_qt/components/chat_panel.py` *was* run once this session (to test the old behavior) and immediately reverted **all** uncommitted changes to that file, exactly the same mistake documented in session 07. Caught immediately and fixed via `git stash pop`, restoring everything with no lost work - see "Errors and fixes" below for the reinforced lesson.
11. Full suite (`pytest tests/ --ignore=tests/test_katch_config.py --ignore=tests/test_katch_keyword_matcher.py`): **167 passed** (165 prior + 2 new dimming tests; same 2 pre-existing unrelated katch-module failures excluded, untouched).
12. User confirmed the fix works ("that worked").

## Key Decisions

- **Don't use `QGraphicsEffect` on a widget that has a child with its own separate `QGraphicsEffect`.** This was the core lesson of the regression: Qt's effect compositing for widget subtrees doesn't reliably nest two independently-installed `QGraphicsEffect`s (parent + child) - the child's own effect-rendered output can get dropped/suppressed by the parent's effect capture. Prefer baking visual changes (opacity, color) directly into pixel data or per-property stylesheet overrides instead, when a widget subtree already has an effect anywhere in it.
- **Offscreen verification must load the real QSS stylesheet when the property under test is styled via QSS.** The first (buggy) implementation was "verified" via a script that only checked a bare `QApplication` with no stylesheet loaded - which meant the title's background-color (entirely QSS-driven) was never actually being tested, letting the regression through. All dimming tests/scripts in this session were rebuilt to explicitly load `dark.qss` (or `light.qss`) the same way `MainWindow._apply_theme()` does, including the `__ASSETS_DIR__` token substitution.
- **`git stash push -- <file>` reverts every uncommitted change to that file, not just the most recent one** - reconfirmed the hard way this session, exactly as documented in session 07. When you need to test "does this fail without my last change," reproduce the old behavior in an isolated scratch script/temporary in-file edit instead of stashing a file with other uncommitted work sitting in it.
- Kept the dimming toggle wired into the existing `ChatPanel.set_streaming(active)` method rather than adding new call sites in `stream_gui.py`, since that method already fires at exactly the right moments (stream start/finish/error).
- Removed `chat_panel.clear_preview()` from `_on_stream_started` rather than adding a separate "restore after clear" path - simpler to just never clear it and let dimming handle the visual change.

## Files Changed

- `gui_qt/components/chat_panel.py`:
  - `_LandscapePreviewLabel`: added `_dimmed` state, `set_dimmed(bool)`, `_dim_pixmap()` (grayscale + `QPainter`-composited fade baked into pixel data), and `_DIMMED_TITLE_STYLE` (inline stylesheet override for the title overlay while dimmed). No `QGraphicsEffect` added to this widget.
  - `ChatPanel`: added `set_preview_dimmed(dimmed)`; `set_streaming(active)` now also calls `set_preview_dimmed(active)` alongside its existing clip-button toggle.
- `gui_qt/stream_gui.py`: removed the `self.chat_panel.clear_preview()` call from `_on_stream_started` (the preview now persists, dimmed, instead of being wiped).
- `tests/test_chat_panel_preview_dimming.py` (new): loads the real theme QSS; `test_title_overlay_stays_visible_and_styled_through_dim_cycle` (the regression test the user asked for) and `test_preview_image_desaturates_and_fades_while_dimmed`.

## Verification Performed

- Full suite: **167 passed** at the final checkpoint (165 prior + 2 new; same 2 pre-existing unrelated `test_katch_*` failures, not touched).
- Offscreen scripts with the real `dark.qss` loaded confirmed, across a full dim -> undim cycle: title stays visible with its darkened background bar intact and text preserved; image pixel goes fully grayscale (`r == g == b`) while dimmed; both fully restore to original appearance after undimming.
- Confirmed the new regression test actually fails against the old buggy code: reproduced the old `QGraphicsOpacityEffect`-on-parent approach in an isolated scratch script (not touching any tracked file) and observed the title bar pixel render as pure, unmodified red - i.e. no background painted at all - which is exactly what the test's first assertion (`pixel_before != (255, 0, 0)`) is designed to catch.
- **Confirmed live in the real running app by the user** ("that worked") - both the dimming behavior and the restored title overlay.

## Current Status

- Feature complete and confirmed working live: preview thumbnail + title now desaturate and fade while a stream is being watched, and fully restore when it stops. The title-overlay regression introduced partway through this session has been fixed and is now covered by an automated test.

## Things Not Yet Tried / Open Follow-ups

- The specific dimmed opacity (0.55 for the image, alpha 90/140 for the title's background/text) and grayscale treatment were chosen as reasonable defaults, not something the user explicitly tuned - if it feels too subtle or too aggressive in longer real-world use, these are simple constants to adjust (`_LandscapePreviewLabel._DIMMED_OPACITY` and `_DIMMED_TITLE_STYLE`).
- As in session 07: if the selected favorite ever differs from the channel actually being streamed (an edge case not currently reachable through normal UI flow, since streaming is always started from the currently-selected favorite), the dimmed preview shown would represent the previously-selected channel, not necessarily the one now playing. Not an active concern, just noted in case the app's flow changes later.
- Carried over from session 07, still untested: the title overlay's square bottom corners vs. the image's rounded corners, and long-title word-wrap stress testing beyond a couple of lines.
