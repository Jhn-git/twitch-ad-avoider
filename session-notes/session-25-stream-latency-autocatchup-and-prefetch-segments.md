# Session 25: Stream-Start Latency, Automatic Catch-Up, And Twitch Prefetch Segments

Date: 2026-07-19

## What Happened

User reported that starting a stream lands nearly 30s behind the live edge (requiring a manual **Go Live** click), and even after catching up, playback settles back to roughly 7s behind after a while — asked whether that's normal, mentioning they had "HLS live edge = 1" configured in the daily exe build at `C:\Users\<user>\Desktop\Jhn Apps\jhn-twitch-viewer`. Investigated in plan mode (two parallel Explore agents, one on the frontend player, one on the backend stream service), designed and iterated on a fix through several rounds of user pushback on the correction mechanism, implemented, unit-tested, and did a static/JSX smoke test. The user then tested the built fix live and confirmed "that is working much better."

### 1. Root causes found

- **Nothing re-synced to live on startup.** This was already `TODO.md` priority #1. hls.js starts playback ~3 segments behind the true edge by design (`liveSyncDurationCount: 3` in `gui_web/components/video_stage.jsx`), and loading time stacks on top of that before playback even begins. The only existing recovery was the manual Go Live button.
- **The player never actually received Twitch's low-latency stream at all — the real root cause of the "ancient Egypt" baseline latency.** Found by reading the vendored streamlink Twitch plugin (`.venv/Lib/site-packages/streamlink/plugins/twitch.py`): the app's `twitch_low_latency`/`hls_live_edge` settings only take effect inside `TwitchHLSStreamReader`/`Worker` (`twitch.py:265-283`), which only run when `stream_obj.open()` is called. That only happens on the **recording** thread (`web_stream_service.py:576`, for the Clip feature). The **playback** path never calls `.open()` — it just proxies Twitch's raw manifest byte-for-byte (`_proxy_playlist`/`_rewrite_playlist`). So the settings the user believed were controlling player latency were doing nothing for what was on screen, and Twitch's actual low-latency "prefetch" segments (`#EXT-X-TWITCH-PREFETCH` tags, present in the manifest already) were being silently dropped because hls.js doesn't recognize that Twitch-proprietary tag.

### 2. Design process (several rounds of user pushback, useful to preserve)

Initial plan proposed a periodic silent hard-cut supervisor. User pushed back hard on two fronts:
- Didn't want any reintroduction of hls.js's own uncapped auto-speedup (the mechanism behind session-22's Bug B — 2x playback fighting a deliberate scrub-back) — explicitly doesn't want speech to sound sped-up, regardless of the fact that browsers preserve pitch by default on `playbackRate` changes (their objection was to the *unnatural pace*, not pitch distortion).
- Also didn't want a repeating hard seek every couple seconds — worried that would itself feel like "stopping and starting all the time."
- Asked to think outside the box rather than just picking between those two extremes ("It might be all there is by looking inside the box... surely this isnt everything").

That pushback is what led to investigating the streamlink plugin and finding the actual root cause (section 1, second bullet) — the user's instinct that there should be a better answer than "patch around drift after the fact" was correct.

Final design, confirmed via `AskUserQuestion`: build the root-cause backend fix now (not deferred), and cap the client-side correction rate at **~1.1x, barely perceptible** (offered a 1.25x "mildly noticeable" alternative; user chose the milder one).

### 3. Fix implemented

**Phase 1 — client-side catch-up**, all in `gui_web/components/video_stage.jsx`:
- `userSeekedRef` — tracks a deliberate rewind (set in `seekFromEvent`, the scrub-bar drag handler, and in `handleDayTimelineClick`'s actual seek). Cleared by `syncToLiveEdge` and reset whenever `playbackUrl` changes.
- `syncToLiveEdge(video)` — the clean instant seek (same math the old `goLive()` inlined: `buffered.end() - 0.5s`), now shared by the Go Live button and the automatic emergency catch-up.
- `maybeAutoCatchUp(video)` — runs every `timeupdate`/`progress` tick, right after `updateSeekVisuals` (so `isLiveRef` already reflects the current gap). Never fires while `userSeekedRef.current` or `video.paused`. A gap beyond `EMERGENCY_GAP_SECONDS = 15` gets one clean instant seek (covers startup, and any rare pathological stall later). A smaller "not live" gap (per the existing `isLiveRef` hysteresis, `LIVE_EDGE_ENTER/EXIT_SECONDS = 4/10`) gets `video.playbackRate = CATCHUP_PLAYBACK_RATE (1.1)` instead of a visible jump, dropped back to `1` the moment it's caught up or the user takes over.
- Deliberately **did not** re-enable hls.js's own `maxLiveSyncPlaybackRate` (stays `1`/off, per session-22's fix) — this is a separate, self-built rate controller, so it can't reintroduce Bug B.

**Phase 2 — actually deliver Twitch's low-latency segments**, in `src/web_stream_service.py`:
- `_rewrite_playlist` now recognizes `#EXT-X-TWITCH-PREFETCH:<uri>` lines and rewrites them into a standard `#EXTINF:<duration>,` + proxied-URI pair, so hls.js sees them as normal playable segments instead of silently ignoring them. Per the real streamlink parser (`twitch.py:88-116`), the tag's value *is* the segment URI directly (not a partial/byte-range scheme), so this reuses the existing `_proxy_resource`/`_resource_url` fetch machinery as-is — no new fetch code.
- Duration is estimated as a running average of the playlist's regular `#EXTINF` segments, mirroring `TwitchM3U8Parser`'s own approach — chains the same duration across consecutive prefetch entries rather than recomputing.
- Gated behind the existing `twitch_low_latency` config flag (`web_stream_service.py:401`) — this is what finally makes that setting do something for playback, not just recording, and doubles as an instant rollback switch.
- Explicitly **left out of scope**: ad-filtering for the playback path (ads aren't filtered from the browser today either — separate pre-existing gap, not bundled into this fix) and the heavier alternative of fully replacing the raw-manifest proxy with a local HLS-repackaging server built on `TwitchHLSStreamReader`/`Writer` directly (would give the tightest possible latency but is a much larger rewrite — flagged as an escalation path if prefetch-rewriting alone isn't enough, not started).

## Key Decisions

- Built a small, self-contained rate controller (`maybeAutoCatchUp`) instead of re-enabling/retuning hls.js's own `maxLiveSyncPlaybackRate` — avoids any risk of reintroducing session-22's Bug B, and gives exact control over the 1.1x cap the user asked for.
- Attacked the root-cause backend latency via a lightweight rewrite inside the existing raw-manifest-proxy architecture (parsing `#EXT-X-TWITCH-PREFETCH` in `_rewrite_playlist`) rather than the heavier option of fully consuming the stream through streamlink's real `TwitchHLSStreamReader`/`Writer` pipeline for playback — much lower risk, reuses existing fetch code, and gets an instant rollback via the pre-existing `twitch_low_latency` toggle.
- Left ad-filtering for the playback path out of scope even though it was discovered missing along the way (ads aren't filtered from the browser today, only from the recording thread) — a separate, pre-existing gap; not bundled into a latency fix without being asked.
- `EMERGENCY_GAP_SECONDS = 15` and `CATCHUP_PLAYBACK_RATE = 1.1` were chosen as reasonable starting points matching the user's stated preference, not empirically tuned across multiple channels/segment cadences — both are isolated as named constants at the top of `video_stage.jsx` for easy retuning.

2. **Daily-exe low-latency verification, investigated, inconclusive.** User has had `twitch_low_latency` checked in settings "for a while" and has been watching streams with it on — but that predates today's fix (the setting existed before, it just didn't do anything for playback until this session). Checked `C:\Users\<user>\Desktop\Jhn Apps\jhn-twitch-viewer\twitchadavoider.exe`: last built **2026-07-19 16:01**, with settings-saved log entries as recent as **16:34** (`logs\twitch_ad_avoider.log`), so the exe was very likely rebuilt after today's changes and the user has been actively using it since — but this wasn't independently confirmed (didn't ask the user directly whether the rebuild specifically included today's `src/web_stream_service.py` edits, only inferred from timestamps). More importantly: **the app's own log file doesn't capture streamlink's internal logger** (searched the whole file for `low.latency|prefetch|low latency stream` — zero hits, including the "This is not a low latency stream" line streamlink logs when a channel doesn't support it), so there's currently no way to confirm from logs whether any specific watched channel actually broadcast Twitch low-latency prefetch segments, as opposed to the setting simply being on with no prefetch data ever present to rewrite. Phase 2 is a safe no-op either way, and Phase 1 (client-side auto-catch-up) works regardless of LL support — so the positive feedback so far is *at least* attributable to Phase 1, with Phase 2's contribution still unconfirmed.

3. **Ads question — this changed my assessment, worth flagging clearly.** Original claim ("ads aren't filtered from the browser today either") was too strong. Re-investigated the installed streamlink Twitch plugin directly: `disable-ads` (which `_new_streamlink_session` sets to `True`) is defined with `help=argparse.SUPPRESS` and, on searching the entire plugin file, **is never actually read anywhere** — it's a vestigial/dead option in this streamlink version. Real ad-segment detection/filtering (`TwitchHLSStreamWriter.should_filter_segment`, `_is_segment_ad`) exists and runs **unconditionally** in the Reader/Writer pipeline, but — same disconnect as low-latency — that pipeline only runs via `.open()` on the **recording** thread, never on the playback proxy. So there is genuinely no ad-filtering code in this app's browser-playback path. Yet the user reports never seeing an ad. **Could not resolve this contradiction from static code reading alone** — plausible explanations (neither confirmed): ad-break timing simply hasn't coincided with their viewing windows, or something about how streamlink acquires the Twitch access token for this client pattern happens to result in ad-free manifests from Twitch's side (a real technique some tools rely on, but no code in this app does it deliberately). Left open rather than guessed at further.
