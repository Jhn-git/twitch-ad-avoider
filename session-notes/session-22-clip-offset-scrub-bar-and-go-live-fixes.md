# Session 22: Clip Offset, Custom Scrub Bar, And Go-Live Fixes

Date: 2026-07-18

## What Happened

User reported three related playback bugs: (1) the Clip button always captured the last N seconds up to "now," ignoring where the playhead actually was if paused/scrubbed back; (2) scrubbing through the buffered stream was inconsistent, mostly didn't work; (3) occasionally fell far behind live with no reliable way to catch up except manually dragging the scrub bar as far right as possible. Investigated, planned (plan mode, approved), implemented, then iterated through several rounds of real-live-channel verification that surfaced additional bugs beyond the original three. A fourth, distinct bug (clipping after a stream ends) was found near the end of the session, diagnosed, and — per the user's go-ahead — fixed before wrapping up. A reusable test-isolation helper was also built to prevent a repeat of an incident where automated testing collided with the user's real running session (see section 4).

### 1. Root causes found

- **Clip offset**: `WebStreamService.create_clip()` (`src/web_stream_service.py`) computed `start_offset` purely from `elapsed = datetime.now() - session.recording_start_time` — pure wall-clock time since the rolling recording started, with zero awareness of the browser's `<video>` playhead position. The Clip button (`gui_web/components/video_stage.jsx`) called `onClip` with no arguments at all.
- **Scrubbing / catch-up-to-live**: the player relied entirely on the native Chromium `<video controls>` scrub bar. Since the HLS stream is live, this is notoriously unreliable, and there was no custom seek UI and no "jump to live" affordance anywhere in the codebase.

### 2. Fix implemented (per approved plan)

- **Backend clip offset**: threaded a new `behind_live_seconds` parameter through `video_stage.jsx` (`handleClip` reads `video.buffered`/`currentTime` at click time) → `stream_manager.jsx` (`createClip`) → `webapi.py` (`create_clip`) → `web_stream_service.py` (`create_clip`), which now computes `target_end = elapsed - behind_live_seconds` before subtracting `duration_seconds`. `behind_live_seconds == 0` reproduces the old (still-correct) live-edge behavior exactly.
- **Custom seek bar**: replaced reliance on the native scrub bar with a purpose-built one — `.seek-track`/`.seek-fill`/`.seek-thumb` in `video_stage.jsx`, driven by `timeupdate`/`progress` listeners (imperative DOM updates via refs, not `setState` per tick) and pointer-event drag-to-seek (`pointerdown`/`pointermove`/`pointerup`, live-updates `video.currentTime` during drag). Native timeline hidden via `video::-webkit-media-controls-timeline { display: none }` (safe here since WebView2 is Chromium-only), native play/pause/volume/fullscreen left untouched.
- **Go Live button**: new button next to the seek bar; new `skipToLive` icon (`icons.jsx`); new CSS (`index.html`).
- **Tests**: `tests/test_web_stream_service.py` gained two tests (`behind_live_seconds` shifts the `-ss` offset correctly; a very large value clamps to `0.0` instead of going negative). `tests/test_webapi.py`'s `FakeStreamService.create_clip` now records call args, plus a new forwarding test.

### 3. Verification found and fixed three more real bugs (live-testing, not caught by unit tests)

All three were found by writing a click-driven pywebview test harness (per the `pywebview-gui-test` skill's Tier-2 pattern) and running it against `jg_darhk` while genuinely live — none of these would have surfaced from `?demo=1` mode or unit tests, since they're all about real hls.js live-buffer timing behavior.

- **Bug A — Go Live undershot the real edge by ~17s.** First implementation targeted `hls.liveSyncPosition` (chosen deliberately per the plan, to avoid seeking to the literal bleeding edge and stalling). Diagnosed via a focused diagnostic script (temporarily exposing `window.__debugHls`) that `hls.liveSyncPosition` tracks `currentTime` almost exactly during *completely normal, untouched* playback — meaning normal live playback for this stream naturally sits ~17s behind the raw `video.buffered` end. That's much larger than the assumed ~2s-per-LL-HLS-segment cushion (`liveSyncDurationCount: 3`), implying this stream's actual segments are longer (~5-6s) than assumed. User's own screenshots confirmed it independently: Go Live left the thumb well short of the bar's right edge, and dragging further right worked fine with no stutter. **Fix**: both the `isLive` check and the Go Live target now use the raw `video.buffered` end (minus a 0.5s epsilon for the seek target) instead of `hls.liveSyncPosition` — i.e., "live" is now defined as "as far right as the seek bar can go," matching what dragging can already reach.
- **Bug B — scrubbing backward would occasionally play at ~2x speed.** Root cause: `maxLiveSyncPlaybackRate: 1.5` (a pre-existing hls.js config value, not something added this session) tells hls.js to auto-accelerate playback whenever `currentTime` falls behind its live-sync target — a mechanism meant to recover from network-caused lag, but with no way to distinguish that from a deliberate user scrub-back. Once scrubbing actually worked (this session's fix), every backward scrub started fighting this auto-catch-up. **Fix**: set `maxLiveSyncPlaybackRate: 1` (hls.js's default = effectively off), since the explicit Go Live button is now the intentional catch-up mechanism and the automatic one is both redundant and actively hostile to deliberate rewatching.
- **Bug C — the Go Live button flickered on/off during completely normal, untouched live playback.** Root cause: `video.buffered`'s end doesn't grow smoothly — it jumps forward in bursts each time a new segment lands, then holds flat while `currentTime` (which advances continuously) catches up to it. So `bufferedEnd - currentTime` naturally sawtooths even while sitting still at live, and the original single `LIVE_EDGE_THRESHOLD_SECONDS = 4` cutoff sat right in the middle of that natural swing, causing false flips on every segment arrival. **Fix**: hysteresis — `LIVE_EDGE_ENTER_SECONDS = 4` (must shrink below this to become "live"), `LIVE_EDGE_EXIT_SECONDS = 10` (must grow past this to become "not live") — so a single segment arrival can no longer cross the whole deadband.

User confirmed after Bug C's fix that it "seems to be working" (not independently re-verified via the automated harness afterward — see incident below for why).

### 4. Incident: automated live-testing collided with the user's own real session

Mid-verification, discovered the user had the real app open (`main.py`, two processes running) on the same channel being used for automated testing. `TEMP_DIR`/`CLIPS_DIR` (`src/constants.py`) are hardcoded relative paths, not per-session-isolated, so the test harness's `_prepare_recording()` hit a real `WinError 32` trying to unlink the user's in-use `temp/recording_jg_darhk.ts`, then (since that error is caught and swallowed) proceeded to open the same file in append mode alongside the user's own session — i.e., **two independent processes briefly wrote to the same recording file concurrently**, risking interleaved/corrupted TS data in the user's real session's recording for roughly the last minute of that test run. Flagged this to the user immediately, recommended they restart their stream to get a clean recording file, and **stopped all further automated live-testing against their real session** for the rest of this session — every fix after that point (Bugs B and C, and the diagnosis below) was verified only by the user directly in their own running app, not by a competing harness.

**This was never explicitly confirmed resolved** — worth checking next time whether the user actually restarted their stream after that warning, and whether any clip made in that window came out corrupted.

### 5. New bug reported, root-caused, and fixed: clipping after the stream ends

After a stream ends, `stream.playback_url` and `stream.recording` are **not cleared** by the natural "ended" transition in `WebStreamService.get_state()` (only an explicit `stop()` clears the session) — so the video element, seek bar, and Clip button all stay mounted/enabled, and the user can keep scrubbing through whatever was buffered before the stream ended. That's already working as the user wants. But clicking Clip after scrubbing mid-way through that cached buffer instead clipped from near the end of the stream, ignoring the scrub position — screenshot showed the user scrubbed to roughly the buffer's midpoint and the clip came out wrong.

**Root cause**: `create_clip()`'s `elapsed = datetime.now() - session.recording_start_time` assumed the recording had been growing continuously at real-time pace for that entire wall-clock span. That's true while live, but once the stream truly ends, the recording *file* stops growing (no more chunks arrive) while `elapsed` kept inflating for however long the user spent afterward browsing/scrubbing before clicking Clip. So by clip time, `elapsed` overshot the file's real content length by however many minutes had passed since the stream actually stopped — which skewed `target_end = elapsed - behind_live_seconds` toward a stale, too-recent "now," landing the clip too close to the end regardless of a correctly-computed `behind_live_seconds`. This is a distinct root cause from the original bug #1 (which was about *not knowing* the playhead position at all) — this one was about the backend's "now" reference itself becoming wrong once real recording had stopped but time kept passing.

**Fix implemented**: `web_stream_service.py::create_clip()` now uses the recording file's own last-modified timestamp (`file_stat.st_mtime`, via `datetime.fromtimestamp`) instead of `datetime.now()` for the `elapsed` calculation. That timestamp only advances when data is actually written, so it tracks `datetime.now()` almost exactly during active live recording (chunks arrive continuously), but correctly freezes at the true stream-end moment once writes stop, no matter how long the user takes afterward to click Clip. The existing `source_path.stat()` call (already used for the `st_size` "not ready yet" check) now captures the full `stat()` result once and reuses it for `st_mtime` too. This directly enables the user's stated want — "I'd like to be able to go back and clip things as well" — from the cached post-end buffer.

New regression test (`tests/test_web_stream_service.py::test_create_clip_uses_file_mtime_not_wall_clock_after_stream_ended`): backdates `recording_start_time` by 700s, backdates the recording file's mtime to `start_time + 400s` (simulating a stream that ran 400s then ended, with 300s having passed since), and confirms the `-ss` offset is computed from the 400s figure, not the 700s wall-clock figure. **Not yet re-verified live** — only unit-tested, since the user was wrapping up for the night; a live post-stream-end clip should still be tried next time it comes up naturally.

### 6. Built: reusable test-isolation helper (`scripts/gui_test_support.py`)

Per the user's request, turned the "recommended skill" from earlier in this session into an actual small module rather than just a suggestion. Provides:
- `warn_if_app_running()` — checks (Windows-only, via `Get-CimInstance`) for other running `main.py` processes and prints a loud warning if found, without aborting.
- `patch_temp_dir(scratch_dir)` — a context manager that monkeypatches `src.web_stream_service.TEMP_DIR` (the module-level name that `_prepare_recording()` actually reads, not `src.constants.TEMP_DIR` directly) to a scratch directory for the duration of a test, restoring it afterward even on error.
- `make_scratch_config(scratch_dir)` — a fresh `ConfigManager` rooted entirely in a scratch directory with `clip_directory` also redirected, so a test run's settings/clips can never touch the user's real files.

Smoke-tested directly (not via a live pywebview session): confirmed `TEMP_DIR` is correctly patched and restored, and that `warn_if_app_running()` runs without error (found zero running instances at that point — the two `main.py` processes seen earlier in the session were no longer running). **Not yet exercised in a real live-channel test run** — next time a live verification session is needed, this should be the first import, used the same way the docstring shows.

### 7. Live-style verification of the post-stream-end clip fix — using real footage, plus a deeper bug found along the way

Picked this back up after a compact/overnight break. Rather than waiting for a real stream to end, found that `temp/recording_jg_darhk.ts` (4.8GB, last written Jul 17 23:28, confirmed via `scripts/gui_test_support.find_running_app_pids()` that no real app instance was running / actively writing it) was sitting idle from a previous live-testing session — exactly the "stream ended, cached recording present" scenario, for free. Built a verification script (not checked into the repo, scratch-only) that:

- Extracted a clean, real 300s segment from the actual recording (read-only — the original file was never opened for writing) into a scratch `.ts` file, using `scripts.gui_test_support.make_scratch_config()` to keep all output confined to a scratch `clips/` directory.
- Ran the *real* `WebStreamService.create_clip()` (real ffmpeg, real subprocess, not mocked) twice against it: once simulating the user scrubbed to the buffer's midpoint (`behind_live_seconds=180`), once simulating they stayed at the live edge (`behind_live_seconds=0`) — with the file's mtime backdated ~900s before "now" to reproduce the exact "browsed for a while after the stream ended" scenario the bug was about.
- Confirmed both clips were valid, ~30s, and had different content (not byte-identical).
- Extracted stills from both real output clips and from the original recording at the two independently-computed expected real-timeline positions, and visually compared them: the "midpoint" clip showed the same in-match gameplay frame (down to the exact "SUB GOAL 862/1000" webcam overlay text) as the reference frame at the expected offset, and the "live edge" clip showed the same lobby/character-select frame as its own, different, expected offset. (An automated pixel-checksum comparison was tried first and gave false negatives — almost certainly a container/color-range remux artifact from the double-seek chain, not a real mismatch — so this was confirmed visually instead, which was unambiguous.)

**Result: PASS.** This is now live-verified (well, real-footage-verified) with high confidence, not just unit-tested.

**Bug found along the way (not yet fixed, needs a decision):** while picking an extraction offset, discovered `temp/recording_jg_darhk.ts` has a Windows creation time of **Jul 9** but a last-write time of **Jul 17 23:28** — an 8-day span — while `ffprobe` reports its actual playable content is only **~5.55 hours (19993s)**. Also, `ffmpeg` reported `Non-monotonic DTS` errors and a `Found duplicated MOOV Atom` warning partway into the file. Root cause: `_prepare_recording()` (`web_stream_service.py`) tries to `unlink()` the stale recording file at the start of every new session, but if that fails for *any* reason (`except OSError as exc: logger.warning(...)` — file locked, in use, permissions, anything), it silently proceeds to open the *same old file* in append mode (`open(session.recording_path, "ab")`) while still assigning a **fresh** `recording_start_time = datetime.now()`. That fresh start time no longer corresponds to byte offset 0 of the file — there's old, unrelated content from a previous session sitting at the front of it. Since `elapsed`/`start_offset` math is entirely based on "time since `recording_start_time`," **every clip made in a session that inherited a stale, not-fully-deleted recording file would be measuring from the wrong zero point** — a separate, likely-longstanding bug from the one just fixed, and a plausible explanation for other "clip missed the mark" symptoms beyond the two already diagnosed this arc. Not fixed yet — needs a decision on the right fix (e.g., fail loudly and pick a uniquely-named file instead of silently appending; retry-with-backoff like the file-lock pattern already documented in the `pywebview-gui-test` skill; or something else) rather than guessing at the user's preference.

## Key Decisions

- Chose raw `video.buffered` end (not `hls.liveSyncPosition`) as the reference for both "is live" and the Go Live seek target, once live testing showed `liveSyncPosition` undershoots badly for this stream's actual segment cadence. This may not generalize to every channel/segment-duration combination, but matches what the user can already reach by dragging, which is the more important consistency property.
- Disabled hls.js's automatic live-catch-up playback-rate acceleration entirely (`maxLiveSyncPlaybackRate: 1`) rather than trying to make it coexist with deliberate scrubbing (e.g., by only enabling it near the live edge) — simpler, and the explicit Go Live button already covers the legitimate "help me catch up" need without hls.js silently deciding when to intervene.
- Used hysteresis rather than just raising the single threshold for the flicker fix — more robust to segment-duration variance across different channels than picking one bigger magic number tuned to `jg_darhk` specifically.
- Stopped automated live-testing entirely once it was found to be colliding with the user's real session, rather than trying to work around the shared-file issue in the moment (e.g., by finding and using a scratch `TEMP_DIR` override) — prioritized not risking further interference over finishing out the verification loop for its own sake.
